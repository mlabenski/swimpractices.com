## Subcollection: `users/{userID}/practices` (Practice Recaps)

Stores completed workout recaps for each user. Each document represents a single completed practice session with performance metrics, stroke data, and progress tracking.

### Path

`users/{userID}/practices/{recapID}`

### Schema

| Field | Type | Description |
|-------|------|-------------|
| `practiceID` | `string` | ID of the practice template that was performed |
| `timestamp` | `Timestamp` | When the workout was completed |
| `duration` | `number` | Total workout duration in seconds |
| `distance` | `number` | Total distance swum (meters) |
| `strokes` | `Array<Stroke>` | Breakdown of strokes performed |
| `heartRateAvg` | `number?` | Average heart rate during workout (optional) |
| `favorite` | `boolean?` | Whether user marked this recap as a favorite |
| `practiceName` | `string?` | Name of the practice template |
| `practiceType` | `string?` | Type of practice (e.g., "Sprint", "Endurance") |
| `primaryStroke` | `string?` | Primary stroke of the practice |
| `isIntervalBased` | `boolean?` | Whether the practice uses interval timing |
| `actualDistanceUnit` | `string?` | Unit for actual distance (typically "meters") |
| `plannedDistance` | `number?` | Expected total distance from practice template |
| `plannedDistanceUnit` | `string?` | Unit for planned distance ("yards" or "meters") |
| `plannedDurationSeconds` | `number?` | Expected duration from practice template |
| `totalStrokeCount` | `number?` | Total stroke count during workout |
| `averagePacePer100` | `number?` | Average pace per 100m/y in seconds |
| `strokesPer100` | `number?` | Average strokes per 100m/y |
| `distanceCompletionPercentage` | `number?` | Ratio of actual to planned distance (0-1+) |
| `plannedStrokeDistribution` | `Array<Stroke>?` | Expected stroke breakdown from template |
| `completedSetIndex` | `number?` | Last set index completed (0-based) |
| `completedExerciseIndex` | `number?` | Last exercise index completed (0-based) |
| `completedRoundIndex` | `number?` | Last round index completed (0-based) |
| `plannedExerciseCount` | `number?` | Total exercises in the practice template |
| `plannedRepeatCount` | `number?` | Total repeats in the practice template |
| `completedExerciseApproximation` | `number?` | Estimated number of exercises completed |
| `healthKitWorkoutID` | `string?` | UUID of the associated HealthKit workout |
| `strokeStyleLaps` | `Array<StrokeStyleLap>?` | Per-lap stroke style data from HealthKit |

### Notes

- Documents use auto-generated Firestore IDs (allows multiple completions of the same practice template)
- Saving a recap also increments `practiceCount` on the parent user document

- The `favorite` field can be updated independently via `updatePracticeRecapFavorite`
- Many fields are optional for backward compatibility with older app versions

## Collection: `analytics`

Contains analytics data including the community leaderboard.

### Schema

#### Document: `leaderboard`

Written by the `calculateLeaderboard` scheduled function (every 24 hours). Aggregates all documents in `users/{userID}/practices` (practice recaps) per user, then stores three computed views plus metadata.

| Field | Type | Description |
|-------|------|-------------|
| `byPracticeCount` | `Array<LeaderboardEntry>` | Top 25 users ranked by number of completed recaps (descending) |
| `byTotalDistance` | `Array<LeaderboardEntry>` | Top 25 users ranked by cumulative recap distance (descending) |
| `highScore` | `HighScore` | Single best single-workout distance across all users |
| `updatedAt` | `Timestamp` | When the leaderboard was last recalculated |

### LeaderboardEntry Schema

Each entry appears in both `byPracticeCount` and `byTotalDistance` (same shape; sort order differs).

| Field | Type | Description |
|-------|------|-------------|
| `userID` | `string` | User ID |
| `username` | `string` | Display name from the user document (`"Unknown"` if missing) |
| `practiceCount` | `number` | Total completed recaps for this user |
| `totalDistance` | `number` | Sum of `distance` across all recaps (meters) |

### HighScore Schema

| Field | Type | Description |
|-------|------|-------------|
| `userID` | `string \| null` | User who logged the longest single workout |
| `username` | `string \| null` | That user's display name |
| `practiceID` | `string \| null` | `practiceID` from the recap, or recap document ID if absent |
| `distance` | `number` | Longest single-workout distance (meters); `0` when no recaps exist |

### Example Document

```javascript
// Path: analytics/leaderboard
{
  byPracticeCount: [
    {
      userID: "user1",
      username: "Alice Smith",
      practiceCount: 42,
      totalDistance: 85000
    },
    {
      userID: "user2",
      username: "Bob Jones",
      practiceCount: 38,
      totalDistance: 72000
    }
  ],
  byTotalDistance: [
    {
      userID: "user1",
      username: "Alice Smith",
      practiceCount: 42,
      totalDistance: 85000
    },
    {
      userID: "user3",
      username: "Carol Lee",
      practiceCount: 15,
      totalDistance: 91000
    }
  ],
  highScore: {
    userID: "user3",
    username: "Carol Lee",
    practiceID: "practice789",
    distance: 12000
  },
  updatedAt: Timestamp
}
```

### Notes

- Each ranked array is capped at 25 entries (`LEADERBOARD_LIMIT`).
- Only users with at least one recap are included.
- Distances come from recap `distance` fields (meters), not from `practiceTracker`.
- The legacy `leaderboard` array field (flat `{userID, totalDistance}` list) is no longer written by `calculateLeaderboard`.

Same schema as `practices` collection.

## Data Flow

1. **User Onboarding:**
   - User submits widget responses → `widget_responses` collection
   - Trigger updates user settings in `users` collection

2. **Practice Generation:**
   - User requests practice → Cloud Function
   - Function reads user data and recent practices
   - AI generates new practice
   - Practice saved to `practices` collection
   - Practice ID added to user's `practices` array
   - Practice saved to `last_practice` for future context

3. **Practice Statistics:**
   - New/updated practice → Trigger calculates stats
   - Stats (totalYards, statistics) saved back to practice document

4. **Practice Completion (Recaps):**
   - User completes a workout on Apple Watch or iPhone
   - `PracticeRecapRepository` saves recap to `users/{userID}/practices` subcollection
   - Also increments `practiceCount` on the user document
   - Offline recaps are cached locally and synced when connectivity returns
   - Recaps can be queried by date range or fetched in full (ordered by timestamp descending)

5. **Leaderboard:**
   - Users complete workouts → recaps saved to `users/{userID}/practices`
   - `calculateLeaderboard` (every 24 hours) scans all user recaps
   - Writes `byPracticeCount`, `byTotalDistance`, `highScore`, and `updatedAt` to `analytics/leaderboard`

## Notes for AI Training

### Key Patterns

1. **Document IDs:**
   - All documents include their Firestore document ID in an `id` field
   - This is set after creation: `await docRef.update({id: docRef.id})`

2. **Timestamps:**
   - Created/updated timestamps use `admin.firestore.Timestamp.now()`
   - Serialization requires special handling (see `helpers/serializers.js`)

3. **Arrays:**
   - User practices stored as array of document IDs (not subcollection)
   - Array updates use `admin.firestore.FieldValue.arrayUnion()`

4. **Nested Structures:**
   - Practices have deeply nested structure (practice → sets → exercises)
   - All levels have sequential IDs starting at 0

5. **Auto-generated Fields:**
   - `tags` auto-generated on practice creation
   - `totalYards` and `statistics` auto-calculated on write
   - IDs auto-assigned if missing

### Common Operations

- Reading user data: `db.collection("users").doc(userId).get()`
- Querying practices by user: `db.collection("practices").where("userID", "==", userId)`
- Batch operations for leaderboard: Use `db.batch()`
- Scheduled cleanup: Clear collection before adding new documents

### Error Handling

- Check document existence before reading: `if (!doc.exists)`
- Graceful fallbacks for missing user data
- Try-catch for async operations with detailed logging
- Never throw errors in Firestore triggers (log and return null)
